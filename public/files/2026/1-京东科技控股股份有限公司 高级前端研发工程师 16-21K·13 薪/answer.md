## 一、项目深度深挖

### 1. 虚拟滚动优化

**Q: 请你讲一下虚拟滚动的实现原理，你在实现过程中遇到了哪些坑，怎么解决的？**

A:
虚拟滚动的核心原理就是**只渲染可视区域内的行，不在可视区域的行不创建 DOM**，这样不管十万行还是一百万行，实际渲染的 DOM 数量始终保持在一个固定的小范围，解决了大型表格渲染卡顿问题。

我的实现思路是：
1. 计算可视区域能容纳多少行 → 根据行高算出渲染起始索引和结束索引
2. 只截取这个区间的数据渲染
3. 创建一个占位的大 div 撑起整个滚动区域的高度，让滚动条看起来是完整的
4. 监听 scroll 事件，根据滚动位移重新计算当前应该渲染哪些行

我遇到的几个坑：
- **滚动快速滚动时出现空白**：因为需要重新计算和渲染，有延迟。解决方法是提前多渲染几行（缓冲区），在可视区域上下都多渲染几条，滚动的时候就不会看到空白了。
- **点击、选择等操作需要滚动到指定位置**：需要通过计算得到滚动位置，手动设置 scrollTop。
- **数据更新后保持滚动位置**：需要记住之前的滚动偏移，数据更新后再恢复。

**Q: 如果表格有不同行高怎么办？**

A: 如果行高不固定，需要提前预估每个行的高度，存在一个数组里，渲染后拿到实际高度再更新预估高度。滚动时用累计高度计算位置。这样会有误差，需要动态调整。或者用 `ResizeObserver` 监听渲染后的实际高度，实时更新。

**Q: 除了虚拟滚动，你还做了哪些优化让渲染更快？**

A: 除了虚拟滚动减少 DOM 数量，在 React 层面我做了：
- 使用 `useMemo` 缓存计算结果，避免每次渲染都重新计算
- 使用 `React.memo` 缓存行组件，只有数据变化时才重新渲染
- 把数据扁平化，减少对象嵌套访问，提升访问速度
- 懒加载非关键数据，比如展开详情才加载额外信息

### 2. Electron 项目迁移到 Web

**Q: 你把一个 Electron 项目迁移到 Web 端，具体做了哪些工作？遇到了哪些兼容性问题？**

A:
原来项目是 Electron + React，需要同时保留桌面端和 Web 两个版本，所以我的做法是：
- 用 **git submodule** 把共享的前端代码托管，两个项目（Electron 和 Web）都引用同一个 submodule，这样修改一次代码两边都能更
- 使用环境变量 `process.env.IS_ELECTRON` 区分当前运行环境，不同环境写不同的兼容代码
- 路由方面：Electron 原本没有 URL 路由概念，迁移到 Web SPA 后，我用 react-router 把每个页面组件映射到对应的 URL，同时兼容原来的跳转方式
- Electron API 兼容：原来用了很多 Node.js API 和 electron ipc，我做了一个适配层，Web 环境下用冗余的空函数或者 Web 版本的实现代替，保证编译能过、运行不报错

遇到的兼容性问题主要：
- Node 核心模块不能用了 → 需要找 Web 版本替代，或者自己简单实现
- 文件系统操作不同 → Electron 可以直接读本地文件，Web 需要用 File API，做了适配
- 样式问题 → Electron 用的是固定像素，Web 需要适配不同屏幕

**Q: 为什么不用 monorepo？**

A: 主要因为历史原因，原来两个项目已经是分开的仓库，迁移改动太大。git submodule 能满足我们当前的需求，改动小，风险低。如果重新来的话，我会考虑用 monorepo 管理，更清晰。

### 3. WebSocket 数据压缩优化

**Q: 你自己写了 json-string-pack 工具，将 9MB 压缩到 500KB。讲讲你的压缩思路是什么？**

A:
场景是这样的：我们通过 WebSocket 实时传输大量测试数据，这些数据是很多个相同结构的对象数组，每个对象都有相同的 key。比如 `[{name: 'a', value: 1}, {name: 'b', value: 2}]`，每次都重复传输 `name` 和 `value` 这两个 key，非常浪费空间。

我的压缩思路：
1. **第一次传输时把所有 key 提取出来做字典映射**，比如 `{0: 'name', 1: 'value'}`
2. **之后每个对象只用索引代替 key**，比如 `[[0:'a', 1:1], [0:'b', 1:2]]`
3. 传输过来之后再根据字典把 key 还原回去
4. 这样大量重复的 key 就只传一次，节省了很多空间

为什么不直接用 protobuf？因为我们有些字段是动态的，结构不固定，protobuf 需要预先定义 schema，灵活度不够。所以我这个方案针对**不确定结构但 key 重复率高**的场景效果很好。

实际效果是 9MB 压缩到 500KB，压缩比超过 18:1。对于固定结构的数据我们还是用 protobuf，能再减少 70% 以上的数据量。

### 4. 前端开发规范设计

**Q: 你引入了 TypeScript 和 ESLint 做规范，讲讲你是怎么推动团队落地这个规范的？**

A:
我是逐步推进的，没有一下子全改，那样风险太大：
1. **首先接入 ESLint**：先只做提示，不强制报错，让大家慢慢适应
2. **然后在 git pre-commit 加检查**：只有你修改的文件才检查，不检查整个项目，速度快不影响
3. **TypeScript 也是逐步迁移**：新增文件直接用 ts，老文件不改，修改老文件的时候顺便改成 ts
4. **react 类组件迁移**：同样策略，修改哪个组件就顺便改成函数组件 + hooks，不一次性全改

这样团队接受度高，不会一下子增加很多负担，逐步就能把整个项目规范起来。

## 二、技术基础 & 深度

### HTML/CSS

**1. HTML5 新增了哪些特性？**

A:
- 语义化标签：header、footer、nav、article、section 等
- 表单增强：input 的新类型 email、date、range 等
- 图形：Canvas、SVG
- 多媒体：video、audio
- 存储：localStorage、sessionStorage、IndexedDB
- 其他：WebSocket、Web Worker、History API、Geolocation 等

**2. CSS 盒模型是什么？box-sizing 区别？**

A:
- content-box：width = content 宽度，padding 和 border 会额外往外扩
- border-box：width = content + padding + border，padding 和 border 算在 width 里面，布局好算

现在一般都用 `box-sizing: border-box` 统一处理。

**3. 三栏布局，左右固定，中间自适应，有几种方法？**

A:
1. flex：左右设固定宽度，中间 `flex: 1`
2. grid：`grid-template-columns: 200px 1fr 200px`
3. 浮动：左右 float，中间 margin 留出空间
4. 绝对定位：左右 absolute 定位，中间 margin 留出空间
5. 圣杯布局/双飞翼布局：浮动 + 负 margin

现在一般用 flex 或 grid 最简单。

**4. 什么是重绘和重排？怎么减少重排？**

A:
- 重排（reflow）：DOM 改变影响了布局，浏览器重新计算布局
- 重绘（repaint）：只是样式变了，不影响布局，重新绘制

减少重排：
- 避免逐个修改 DOM 样式，可以统一改好再一次性设置
- 用文档片段（DocumentFragment）批量插入 DOM
- 把 DOM 设置为 display: none 后修改，改完再显示（只触发两次重排）
- 避免使用 table 布局，小变化都会触发整个 table 重排
- 动画用 GPU 加速，用 transform 不要改 left/top

### JavaScript

**1. 闭包的理解？应用场景？**

A:
闭包就是函数能够访问其外部作用域变量，即使这个函数在定义它的作用域之外执行。本质上是因为函数创建时就保存了作用域链，所以执行时还能找到原来的变量。

应用场景：
- 函数柯里化
- 模块化（模块化本质就是闭包封装私有变量）
- 防抖节流
- 保存私有状态

**2. 原型和原型链？**

A:
每个函数都有 `prototype` 属性，指向原型对象。每个对象都有 `__proto__` 属性，指向它构造函数的原型。原型对象也有自己的原型，这样一层层链接起来就是原型链。查找属性的时候，先找对象自身，找不到就顺着原型链往上找，直到 null。

**3. Event Loop 机制？宏任务和微任务？**

A:
JS 是单线程，异步任务需要放到任务队列。执行栈空了就会去任务队列取任务执行。

- 宏任务：script（整体代码）、setTimeout、setInterval、I/O、DOM 事件、requestAnimationFrame
- 微任务：Promise.then/catch/finally、process.nextTick、MutationObserver、queueMicrotask

执行顺序：执行完当前宏任务 → 清空所有微任务 → 进入下一个宏任务 → 循环

**4. 深拷贝需要注意哪些问题？**

A:
- 循环引用：需要用 WeakMap 存已经拷贝过的对象，遇到就直接返回，避免死循环
- 不同类型：需要区分数组、对象、Date、RegExp、Function 等不同类型分别处理
- 原型链：一般不需要拷贝原型，保持引用就行
- 特殊对象：比如 Map、Set 也要特殊处理

### React

**1. React Fiber 架构解决了什么问题？**

A:
原来的 React 同步渲染，一旦开始渲染就不能中断，如果组件很大，会阻塞主线程，导致交互卡顿。Fiber 把渲染任务拆分成多个小任务，渲染过程中可以暂停，把主线程让给优先级高的任务（比如用户输入），做完再继续渲染。解决了大渲染阻塞交互的问题。

**2. React Hooks 和类组件的区别？**

A:
- hooks 用函数组件，代码更简洁，不需要处理 this 的问题
- 逻辑复用方便，自定义 hooks 比 HOC 或者 render props 更干净
- 生命周期概念被替代，用 useEffect 处理副作用
- 缺点是需要理解 hooks 依赖规则，容易遇到闭包坑

我现在开发都用函数组件 + hooks，推荐团队这么用。

**3. React 性能优化手段有哪些？**

A:
- 减少不必要渲染：React.memo、useMemo、useCallback
- 懒加载：React.lazy + Suspense 路由懒加载
- 虚拟滚动：大数据列表
- 缓存：useMemo 缓存计算结果
- 不可变数据：不要直接修改 state，总是返回新对象
- key 的正确使用：列表渲染不要用 index 当 key，用稳定唯一 id

### 浏览器 & 网络

**1. 从输入 URL 到页面渲染完成，整个过程发生了什么？**

A:
1. DNS 解析：把域名转换成 IP 地址
2. 建立 TCP 连接：三次握手
3. 发送 HTTP 请求
4. 服务器处理请求，返回响应
5. 浏览器拿到响应内容，开始解析 HTML
6. 构建 DOM 树、CSSOM 树，合成渲染树
7. 布局（计算位置大小）→ 绘制（绘制到屏幕）
8. 遇到 JS 会阻塞解析，因为 JS 可能修改 DOM/CSSOM
9. 图片等资源加载完成后，页面完成加载

**2. 浏览器缓存机制？**

A:
- 强制缓存：响应头 Cache-Control/Expires，直接从缓存读，不发请求
- 协商缓存：强制缓存失效后，发请求给服务器，服务器判断是否修改，没修改返回 304，用缓存

**3. 跨域怎么解决？**

A:
- CORS：后端设置 Access-Control-Allow-Origin，最简单
- JSONP：利用 script 标签不受同源策略限制，只支持 GET
- 反向代理：开发环境用 webpack-dev-server 代理，生产环境 Nginx 配置
- postMessage：跨页面、iframe 通信

## 三、工程化 & 性能优化

### Webpack

**1. Webpack 打包流程？**

A:
1. 初始化：从配置文件读取入口和插件
2. 编译：从入口开始，递归找出所有依赖模块
3. 构建：每个模块做对应的转换（比如 loader 处理 ES6 -> ES5，less -> CSS）
4. 输出：把打包好的模块合并成 chunks，输出到文件

**2. 怎么提升打包速度？**

A:
- 多线程：thread-loader 或者 parallel-webpack
- 缓存：cache-loader 或者 webpack5 自带缓存
- 缩小搜索范围：include/exclude 排除 node_modules
- 代码分割：splitChunks 拆分第三方库，缓存利用
- 开发环境不用 source-map，或者用 cheap-module-eval-source-map

**3. Tree Shaking 是什么？**

A:
摇树，就是把没有用到的代码删掉，不打包进去。依赖 ES Module 的静态分析，只有 ESModule 能做，因为它是静态导入的。webpack 默认支持，生产环境会自动开启。

### 性能优化

**1. 首屏加载慢怎么优化？**

A:
- 网络层面：gzip/brotli 压缩，CDN 分发，HTTP2
- 资源层面：图片懒加载，图片格式用 webp/avif，小图用 base64，雪碧图，异步加载 JS
- 代码层面：路由懒加载，代码分割，Tree Shaking，SSR 服务端渲染
- 缓存：合理设置 HTTP 缓存，利用浏览器缓存

**2. 性能指标有哪些？**

A:
- FP（First Paint）：首次绘制，画第一个像素
- FCP（First Contentful Paint）：首次内容绘制，绘制出第一个文本/图片
- LCP（Largest Contentful Paint）：最大内容绘制，代表首屏主要内容加载完成，一般要求小于 2.5s
- FID（First Input Delay）：首次输入延迟，衡量交互响应性
- CLS（Cumulative Layout Shift）：累积布局偏移，衡量页面稳定性

## 四、岗位匹配问题

### IM & Electron 相关

**1. 如果让你做一个 IM 客户端，你怎么架构设计？**

A:
我大概会分成几层：
- **网络层**：封装 WebSocket 长连接，处理心跳、断网重连、消息重发
- **数据层**：维护消息列表、会话列表，用 redux 或 zustand 管理状态，本地缓存做持久化
- **组件层**：抽离消息气泡、输入框、会话列表等通用组件
- **业务层**：处理不同消息类型（文本、图片、语音、文件）的展示和交互

性能方面：
- 消息列表用虚拟滚动，因为聊天记录可能很多
- 图片预加载、懒加载，缩略图先显示
- 消息分片加载，滚动到底部再加载更早的记录

**2. IM 长连接 WebSocket 有什么坑？**

A:
- **断网重连**：网络切换、休眠唤醒都会断开，需要监听断开事件，定时重试，用指数退避（越来越久重试一次，避免服务器压力）
- **心跳机制**：定期发心跳包，检测连接是否存活，因为有时候网络断开 TCP 不会触发 close 事件
- **消息乱序**：网络延迟可能导致后发的先到，需要序号排序
- **消息丢了怎么办**：需要做消息确认，没 ACK 就重发

**3. Electron 主进程和渲染进程怎么通信？**

A:
通过 ipcMain 和 ipcRenderer 通信。主进程用 ipcMain.on 监听，渲染进程用 ipcRenderer.send 发消息。需要注意避免阻塞，大数据传输可以用共享内存或者文件。

## 五、算法

### 1. 两数之和

方法：用哈希表存值和索引，遍历一遍，每次找 target - current 是否在哈希表里，有就返回，没有就存进去。时间复杂度 O(n)，空间 O(n)。

### 2. 快速排序

思路：选一个基准，把小于基准的放左边，大于放右边，然后递归排序左右两边。平均时间 O(nlogn)，最坏 O(n²)（当数组已经有序选第一个当基准时）。优化：随机选基准，或者三数取中。

### 3. 判断链表是否有环

方法一：快慢指针，快指针走两步，慢指针走一步，如果有环一定会相遇。空间 O(1)。
方法二：用 Set 存访问过的节点，遇到重复就是有环。空间 O(n)。
